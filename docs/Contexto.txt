Esse projeto está sendo desenvolvido dentro de uma fintech cujo principal produto é o crédito consignado, e o objetivo é substituir completamente um conjunto de processos operacionais que hoje são manuais, fragmentados e altamente dependentes de pessoas específicas. Esses processos são, basicamente, três grandes blocos que hoje funcionam de forma pouco integrada: o envio de remessas (que é o processo de mandar os contratos para desconto), o registro de baixas (quando os pagamentos efetivamente acontecem) e a conciliação (que valida se o que foi esperado bate com o que foi pago). Embora o escopo final envolva esses três módulos, neste momento o foco está concentrado na coleta das datas de corte e na produção das remessas, porque são as etapas mais críticas e que geram maior risco operacional.

Hoje a empresa trabalha com cerca de 26 processadoras e aproximadamente 77 convênios, sendo que esse número não é fixo — pode aumentar ou diminuir conforme novos contratos são firmados ou encerrados. Cada convênio é praticamente um “mundo próprio”: ele pode ter regras diferentes, aceitar produtos diferentes (crédito, benefício, compras), exigir layouts específicos de arquivo, ter códigos próprios de averbação e até formas diferentes de envio. Isso significa que qualquer sistema que a gente construa não pode assumir padronização — ele precisa ser flexível por natureza. No estado atual, essas diferenças são tratadas manualmente por analistas, muitas vezes usando Excel, com regras implícitas que não estão documentadas formalmente.

O ponto de partida de todo o processo é a data de corte. Essa é provavelmente a variável mais sensível de todo o sistema. A data de corte define até quando uma remessa pode ser enviada para uma determinada processadora em um determinado convênio. Se essa data for perdida, não existe fallback: o arquivo não pode mais ser importado, o desconto não acontece e a empresa fica com o prejuízo direto. Por causa disso, a coleta e o controle dessas datas não são apenas um detalhe operacional — eles são o núcleo do problema.

Atualmente, essas datas não estão centralizadas. Elas ficam distribuídas nos portais das processadoras, e cada uma expõe isso de uma forma diferente. Em alguns casos, existe API para consulta, o que seria o cenário ideal, porque permite uma coleta automatizada e confiável. Em outros casos, não existe API, mas é possível acessar o portal via automação (por exemplo, usando Playwright). E existem ainda casos onde não dá para automatizar, seja por captcha, restrições de acesso ou simplesmente porque o esforço não compensa o volume daquele convênio — nesses casos, a coleta precisa continuar manual.

Existe uma preocupação importante aqui que não é apenas técnica, mas também econômica. Nem sempre faz sentido contratar uma API de uma processadora, porque o custo pode ser maior do que o retorno gerado pelos contratos daquele convênio. Por isso, a estratégia não pode ser única — ela precisa ser híbrida. Parte dos convênios vai usar API, parte vai usar scraping e parte vai continuar manual, e o sistema precisa suportar esses três modos de operação convivendo ao mesmo tempo.

Do ponto de vista técnico, já foi validado que é possível fazer scraping em cenários mais complexos, inclusive aqueles que exigem certificado digital. No caso da ConsigFácil, por exemplo, o acesso é feito via mTLS (certificado cliente), e o problema inicial era que o Playwright não consegue interagir com o popup nativo de seleção de certificado. Isso foi resolvido usando uma policy do Chrome chamada AutoSelectCertificateForUrls, que força a seleção automática do certificado, permitindo que o acesso seja feito diretamente pelo navegador controlado pelo Playwright. Esse detalhe é importante porque mostra que a automação não é trivial e tem limitações reais, como dependência de ambiente Windows e do uso do Chrome real em vez do Chromium padrão .

A ideia para esse módulo de datas de corte é construir um serviço que rode diariamente (provavelmente containerizado), que percorra todas as fontes disponíveis (APIs e scrapers), colete as datas, compare com o histórico já armazenado e identifique qualquer mudança. Quando houver alteração, o sistema deve notificar os analistas responsáveis, provavelmente via e-mail, e ao mesmo tempo atualizar um calendário interno de remessas. Esse calendário não é só informativo — ele vai ser usado pelo módulo de remessas para decidir quando iniciar o processamento. Ou seja, a coleta de datas de corte não é um módulo isolado, ela alimenta diretamente o restante do sistema.

Outro ponto importante é que o sistema precisa lidar bem com falhas. Não dá para assumir que todas as coletas vão funcionar todos os dias. Pode ter portal fora do ar, mudança de layout, erro de autenticação, etc. Então, além de coletar, o sistema precisa registrar o status de cada tentativa, permitir fallback manual e garantir que o analista consiga agir rapidamente quando algo não for coletado. Para isso, a interface deve facilitar o acesso direto ao portal e permitir o registro manual da data, sempre com auditoria (quem inseriu, quando, e de onde veio a informação).

Passando para o módulo de remessas, ele é essencialmente um pipeline de transformação de dados. O processo atual começa com uma consulta no BI para identificar o convênio, a processadora e a data de corte. Depois disso, o analista extrai a base de contratos do BankSoft (geralmente via Excel), aplica uma série de filtros (como exclusão de exonerados e não descontados), ajusta valores quando há desconto parcial e remove contratos que não se encaixam nas regras do convênio. Esse processo depende de bases auxiliares em Excel, como a lista de exonerados e a tabela de descontos parciais, e é feito manualmente com cruzamentos de dados .

Depois dessa preparação, entra uma decisão importante: se a remessa é automática ou manual. Em alguns casos, a própria processadora já faz o processamento automaticamente, e o analista só valida. Em outros, é necessário gerar um arquivo (TXT posicional ou Excel), seguindo um layout específico da processadora, e fazer o upload manual no portal. Cada processadora tem seu próprio padrão de layout, com regras diferentes de preenchimento, códigos obrigatórios e validações específicas. Isso torna a geração do arquivo uma etapa bastante sensível.

Após o envio, a processadora retorna um arquivo de críticas, que precisa ser analisado. Essas críticas podem ser tratáveis (por exemplo, divergência de dados ou necessidade de ajuste) ou não tratáveis (como falecimento ou bloqueio judicial). As tratáveis exigem correção e reprocessamento, enquanto as não tratáveis levam à exclusão do contrato da remessa. Só depois disso o processo é considerado concluído.

O objetivo do sistema é transformar esse fluxo inteiro em algo automatizado, estruturado e rastreável. Em vez de depender de Excel e decisões manuais, a ideia é ter um motor de regras que entenda as particularidades de cada convênio e processadora, aplique os filtros automaticamente e gere os arquivos já no formato correto. Para isso, já existe uma modelagem de dados sendo construída, com entidades como Cliente, Convênio, Processadora, Produto, Contrato, Parcela, Remessa, Crítica e Layout. Essa modelagem é importante porque o sistema não é dono da origem dos dados (que continuam no BankSoft), mas ele precisa organizar e processar essas informações de forma consistente .

Do ponto de vista de arquitetura, a ideia não é construir um monólito, mas sim separar os domínios em serviços independentes. O módulo de datas de corte deve ser um serviço próprio, o módulo de remessas outro, e no futuro o mesmo vale para baixas e conciliação. Esses serviços devem compartilhar informações via banco de dados ou APIs internas, mas precisam ser desacoplados o suficiente para evoluírem independentemente. Isso é importante porque as regras mudam com frequência — novos convênios entram, layouts mudam, processos são ajustados — e o sistema precisa acompanhar essa evolução sem quebrar.

Um ponto que é fácil de subestimar, mas é extremamente importante, é a necessidade de auditoria. Hoje, muitas decisões são tomadas manualmente e não ficam registradas de forma estruturada. No sistema novo, tudo precisa ser rastreável: alterações de data de corte, intervenções manuais, correções de remessa, etc. Isso não é só uma questão de organização, mas também de segurança operacional.

Por fim, o sistema precisa ser construído assumindo que nem tudo será automatizado. Sempre vai existir uma parte manual ou semi-automatizada, seja por limitação técnica, custo ou exceções do processo. O papel do sistema não é eliminar completamente o analista, mas sim reduzir o trabalho manual repetitivo e dar ferramentas para que ele atue de forma mais eficiente e com menos risco.

Esse é o contexto atual do projeto. A expectativa é que qualquer solução proposta leve em conta essa complexidade, evite simplificações excessivas e priorize uma arquitetura que seja robusta, flexível e preparada para crescer junto com o negócio.
